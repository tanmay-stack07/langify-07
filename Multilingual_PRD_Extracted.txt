Core Feature: Live Language Translation — Complete Technical Specification CORE FEATURE This section defines the Live Language Translation feature in exhaustive technical detail. Every subsystem, every API call, every data flow, every state transition, every edge case, and every error scenario is specified here. This section is the definitive reference for all implementation decisions. 3.1 The Concept — How It Works in Plain Language The user opens the Live Translate tab. They see one setting: a dropdown labeled "Translate everything into" with a list of target languages. They select their language — say, English. They click Start Session. The microphone opens. They speak. As the user speaks, the browser captures audio in chunks using the MediaRecorder API. Every 5 seconds of audio is sent to the backend as a binary blob. The backend runs it through Whisper-1, which transcribes the speech and — critically — automatically identifies what language was spoken without any hint. The backend then takes that transcription and sends it to GPT-4o, which translates it into the user&apos;s chosen target language. Both the original text and the translation are returned to the browser and displayed as a card in the live feed. This loop runs continuously until the user clicks Stop Session. At that moment, every utterance from the session is batch-saved to Supabase in a single database transaction. The user can then export the entire session as a formatted document. The user never configured what language they were speaking. They never touched a setting mid-session. They just spoke. The agent did everything else. 3.2 The Two-Model AI Pipeline — The Technical Heart Every single utterance captured during a live translation session passes through exactly two sequential AI API calls. This two-call pipeline is the central technical mechanism of the entire application. Understanding it is understanding the product. PIPELINE — WHAT HAPPENS FOR EVERY UTTERANCE: INPUT: Audio chunk (5 seconds of speech, ~50-300KB, WebM or OGG format) CALL 1 — Whisper-1 (POST /v1/audio/transcriptions) Model: whisper-1 Format: verbose_json Language: NOT PROVIDED — intentionally omitted for auto-detection Output: { text: "transcribed speech", language: "hi", segments: [...] } CALL 2 — GPT-4o (POST /v1/chat/completions) Model: gpt-4o Input: The transcribed text from Call 1 Prompt: "Translate to [target language]. Return only the translation." Output: "Translated text in the target language" RESULT returned to frontend: { originalText, translatedText, detectedLanguage, confidence, timestamp } Why Two Models and Not One? This is an important architectural question that deserves a clear answer for the project report. Whisper-1 is the world&apos;s best open-weight speech recognition model. It is specifically trained on audio and produces state-of-the-art transcriptions. Whisper-1&apos;s built-in /v1/audio/translations endpoint translates to English only — it cannot translate to Marathi, Hindi, Spanish, or any non-English language. This is a hard limitation. GPT-4o is a large language model with exceptional translation capabilities across 100+ languages. It handles nuance, context, tone, and code-switched speech (Hinglish, Manglish) far better than any standalone translation API. But GPT-4o cannot process audio — it only takes text. The combination is therefore not arbitrary. Whisper-1 solves the speech-to-text and language detection problem. GPT-4o solves the text-to-translated-text problem. Neither model can do the full job alone. Together, they produce the best possible result for every utterance, in every language, in both directions. 3.3 Auto-Detection — The Most Critical Design Decision The source language is NEVER sent to Whisper-1. This is the most important architectural decision in the entire application, and it is intentional, deliberate, and non-negotiable. Whisper-1 has built-in language detection that activates when no language parameter is provided. When the model receives audio, it independently identifies what language is being spoken with high confidence. This detection information is included in the verbose_json response alongside the transcription. By omitting the language parameter, the system achieves three things simultaneously: it removes the need for any source language UI, it supports real-time language switching mid-session (the user can speak Marathi, then Hindi, then English within the same session and detection will follow automatically), and it makes the experience feel genuinely intelligent — the system just knows. services/translationService.js — The Complete Two-Model Pipeline const OpenAI = require("openai") const { Readable } = require("stream") const openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY }) async function translateUtterance(audioBuffer, mimeType, targetLanguage) { // ───────────────────────────────────────────────────────── // STEP 1: Whisper-1 — Transcribe + Auto-Detect Language // CRITICAL: No "language" param = automatic detection // ───────────────────────────────────────────────────────── // Convert buffer to a File object that OpenAI SDK accepts const audioFile = new File([audioBuffer], "chunk." + getExtension(mimeType), { type: mimeType }) const whisperResponse = await openai.audio.transcriptions.create({ file: audioFile, model: "whisper-1", response_format: "verbose_json", // gives us language + confidence // language: intentionally NOT set — auto-detection required }) const originalText = whisperResponse.text?.trim() || "" const detectedLanguage = whisperResponse.language // e.g. "hi", "mr", "es" // Calculate confidence from no_speech_prob (0=confident, 1=not speech) const avgNoSpeech = whisperResponse.segments?.length > 0 ? whisperResponse.segments.reduce((sum, s) => sum + (s.no_speech_prob || 0), 0) / whisperResponse.segments.length : 0 const confidence = Math.round((1 - avgNoSpeech) * 100) // Guard: if no speech detected or text too short, skip GPT-4o call if (!originalText || originalText.length < 8) { return { originalText: "", translatedText: "", detectedLanguage, confidence, skippedTranslation: true, reason: "no_speech" } } // ───────────────────────────────────────────────────────── // STEP 2: GPT-4o — Translate to Target Language // ───────────────────────────────────────────────────────── const gptResponse = await openai.chat.completions.create({ model: process.env.TRANSLATION_MODEL || "gpt-4o", temperature: 0.1, // low = accurate and consistent, not creative max_tokens: 600, messages: [ { role: "system", content: buildTranslationPrompt(targetLanguage, detectedLanguage) }, { role: "user", content: originalText } ] }) const translatedText = gptResponse.choices[0]?.message?.content?.trim() || originalText return { originalText, translatedText, detectedLanguage, confidence, skippedTranslation: false, tokensUsed: gptResponse.usage?.total_tokens || 0 } } function getExtension(mimeType) { if (mimeType.includes("webm")) return "webm" if (mimeType.includes("ogg")) return "ogg" if (mimeType.includes("wav")) return "wav" return "mp4" } module.exports = { translateUtterance } 3.4 The GPT-4o Translation Prompt — Engineered for Quality The system prompt sent to GPT-4o is not generic. It is carefully engineered to handle the specific challenges of real-time speech translation — especially code-switched Indian languages, proper nouns, technical terms, and conversational register. services/translationService.js — buildTranslationPrompt function function buildTranslationPrompt(targetLanguage, detectedLanguage) { return `You are a professional real-time speech interpreter. TASK: Translate the following spoken text into ${targetLanguage}. STRICT RULES — follow every rule exactly: Rule 1: Return ONLY the translated text. No preamble. No explanation. No quotation marks around the output. If the user says "Hello" and the target is Hindi, return only: "नमस्ते" Rule 2: Preserve the speaker&apos;s meaning, tone, and register exactly. If the speech is formal, the translation must be formal. If the speech is casual, the translation must be casual. Do not add or remove formality. Rule 3: Code-switched speech handling (CRITICAL for Indian languages). If the speech mixes ${detectedLanguage} with English, translate only the non-${targetLanguage} parts. Keep any ${targetLanguage} words unchanged. Example: "Main kal office jaaunga" → "I will go to the office tomorrow" Example: "Aaj meeting cancel ho gayi" → "Today the meeting got cancelled" Rule 4: Proper nouns, names, and brands are NEVER translated. Keep them exactly as spoken. Example: "Dr. Sharma", "Mumbai", "Google" Rule 5: Technical terms and acronyms keep their original form. Example: "API", "HTML", "COVID-19" remain unchanged. Rule 6: If the source text is already entirely in ${targetLanguage}, return it exactly as-is. Do not modify or paraphrase. Rule 7: Preserve punctuation naturally. Do not add a period if the speech did not end with one. Do not add exclamation marks the speaker did not use. Rule 8: Numbers spoken in the source language should appear as numerals. Example: "teen sau pachaas" → "350" (not "three hundred fifty") ` } Rule 3 is the most critical rule for this application&apos;s primary target audience. Code-switching is not a limitation or an error — it is the natural way tens of millions of Indian professionals speak every single day. The translation pipeline must handle it gracefully, not try to correct it. 3.5 Audio Capture — How Continuous Speech Is Chunked The user does not press a button between sentences. The system captures continuously and automatically sends chunks to the backend. This is implemented using the MediaRecorder API&apos;s timeslice feature, which fires the ondataavailable event at fixed intervals. composables/useMediaRecorder.js — Complete Audio Capture Logic export function useMediaRecorder() { let mediaRecorder = null let stream = null const CHUNK_INTERVAL_MS = 5000 // 5 seconds per chunk const MIN_CHUNK_BYTES = 1000 // discard silence / near-empty chunks async function startCapture(targetLanguage, onChunkReady) { // Request microphone permission — fires permission dialog on first call only stream = await navigator.mediaDevices.getUserMedia({ audio: true }) // Detect the best supported MIME type for this browser // Chrome uses webm/opus, Firefox uses ogg, Safari uses mp4 const mimeType = detectMimeType() mediaRecorder = new MediaRecorder(stream, { mimeType }) // ondataavailable fires every CHUNK_INTERVAL_MS milliseconds mediaRecorder.ondataavailable = async (event) => { // Discard near-empty chunks — this is silence, not speech if (event.data.size < MIN_CHUNK_BYTES) { console.log("[MediaRecorder] Chunk too small, likely silence. Discarding.") return } // Create a Blob with the correct MIME type for the backend const audioBlob = new Blob([event.data], { type: mimeType }) // Notify the session composable that a chunk is ready for processing await onChunkReady(audioBlob, mimeType) } // Start recording. The timeslice parameter = how often ondataavailable fires mediaRecorder.start(CHUNK_INTERVAL_MS) } function stopCapture() { if (mediaRecorder && mediaRecorder.state !== "inactive") { mediaRecorder.stop() } // Stop ALL microphone tracks — this closes the browser mic indicator if (stream) { stream.getTracks().forEach(track => track.stop()) } } function detectMimeType() { const types = [ "audio/webm;codecs=opus", "audio/webm", "audio/ogg;codecs=opus", "audio/ogg", "audio/mp4" ] return types.find(t => MediaRecorder.isTypeSupported(t)) || "audio/webm" } return { startCapture, stopCapture } } Why 5-second chunks? This is the sweet spot between perceived latency and transcription accuracy. Chunks shorter than 3 seconds frequently cut sentences mid-word, which reduces Whisper-1 accuracy significantly. Chunks longer than 10 seconds increase the delay between speaking and seeing the translation, which breaks the "live" feeling. 5 seconds gives the user a translation roughly every 7-8 seconds (5s capture + 2-3s API round trip), which feels responsive while giving Whisper-1 enough audio context to detect language and transcribe accurately. 3.6 The Complete Backend Route — /api/translate This is the most important backend route in the entire application. Every live translation request flows through this route. routes/translation.js — Complete /api/translate Route const express = require("express") const multer = require("multer") const router = express.Router() const rateLimit = require("express-rate-limit") const { translateUtterance } = require("../services/translationService") // Rate limiter: 60 translation requests per IP per hour // This is generous for real use but prevents API abuse const translationLimiter = rateLimit({ windowMs: 60 * 60 * 1000, // 1 hour window max: 60, // 60 requests per window message: { error: "Translation limit reached. Please wait before continuing." } }) // Use memory storage — audio chunks are small (<2MB) // No disk write needed. No cleanup needed. const upload = multer({ storage: multer.memoryStorage(), limits: { fileSize: 2 * 1024 * 1024 }, // 2MB max per chunk fileFilter: (req, file, cb) => { // Accept all audio MIME types Whisper-1 supports const isAudio = file.mimetype.startsWith("audio/") cb(null, isAudio) } }) router.post("/translate", translationLimiter, upload.single("audio"), async (req, res) => { // Validate: audio file must be present if (!req.file) { return res.status(400).json({ error: "No audio chunk received." }) } // Validate: targetLanguage must be present const targetLanguage = req.body.targetLanguage if (!targetLanguage || typeof targetLanguage !== "string") { return res.status(400).json({ error: "Target language is required." }) } const sessionId = req.body.sessionId || null try { // Run the two-model pipeline const result = await translateUtterance( req.file.buffer, req.file.mimetype, targetLanguage ) // If no speech was detected, return a specific signal (not an error) if (result.reason === "no_speech") { return res.json({ success: true, silenceDetected: true }) } // Return the full utterance result return res.json({ success: true, originalText: result.originalText, translatedText: result.translatedText, detectedLanguage: result.detectedLanguage, confidence: result.confidence, skippedTranslation: result.skippedTranslation, targetLanguage: targetLanguage, sessionId: sessionId, timestamp: new Date().toISOString(), }) } catch (error) { console.error("[/api/translate]", error.status, error.message) return res.status(500).json({ error: mapApiError(error) }) } }) function mapApiError(err) { if (err.status === 429) return "AI service is busy. Please wait a moment." if (err.status === 400) return "Audio could not be processed. Speak clearly and try again." if (err.status === 401) return "API authentication failed. Contact the administrator." return "Translation failed. Please check your connection." } module.exports = router 3.7 Frontend Session Management — Complete State Machine The Live Translation session has a precise lifecycle. to Supabase At the end of every session, all utterances are saved to Supabase in a single batch operation. This is the persistence layer of the entire translation feature. routes/session.js — /api/session/end Batch Save const express = require("express") const router = express.Router() const { createClient } = require("@supabase/supabase-js") // Service role key for server-side writes — NEVER use anon key for writes const supabase = createClient( process.env.SUPABASE_URL, process.env.SUPABASE_SERVICE_ROLE_KEY ) router.post("/session/end", async (req, res) => { const { sessionId, targetLanguage, utterances, durationSeconds } = req.body // Validation if (!sessionId || !utterances || !Array.isArray(utterances)) { return res.status(400).json({ error: "Invalid session data." }) } if (utterances.length === 0) { return res.json({ success: true, savedCount: 0, message: "Empty session." }) } try { // INSERT the session record first const { error: sessionError } = await supabase .from("sessions") .insert({ id: sessionId, target_language: targetLanguage, total_utterances: utterances.length, total_duration_seconds: durationSeconds, ended_at: new Date().toISOString(), }) if (sessionError) throw sessionError // BATCH INSERT all utterance rows in a single call const utteranceRows = utterances.map(u => ({ session_id: sessionId, original_text: u.originalText, translated_text: u.translatedText, source_language: u.detectedLanguage, target_language: u.targetLanguage, confidence: u.confidence, created_at: u.timestamp, })) const { error: utteranceError } = await supabase .from("utterances") .insert(utteranceRows) if (utteranceError) throw utteranceError return res.json({ success: true, sessionId, savedCount: utteranceRows.length, }) } catch (error) { console.error("[/api/session/end]", error.message) return res.status(500).json({ error: "Session could not be saved. Please export manually." }) } }) module.exports = router 3.9 Supabase Database Schema — Complete Table Definitions Supabase SQL — sessions table CREATE TABLE sessions ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), created_at TIMESTAMPTZ NOT NULL DEFAULT now(), ended_at TIMESTAMPTZ, target_language TEXT NOT NULL, total_utterances INTEGER NOT NULL DEFAULT 0, total_duration_seconds INTEGER NOT NULL DEFAULT 0, export_url TEXT ); Supabase SQL — utterances table CREATE TABLE utterances ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), session_id UUID NOT NULL REFERENCES sessions(id) ON DELETE CASCADE, created_at TIMESTAMPTZ NOT NULL DEFAULT now(), original_text TEXT NOT NULL, translated_text TEXT NOT NULL, source_language TEXT, -- BCP-47 code from Whisper-1: "hi", "mr", "es" target_language TEXT NOT NULL, confidence FLOAT, -- 0-100, calculated from no_speech_prob duration_seconds FLOAT ); CREATE INDEX idx_utterances_session ON utterances(session_id); CREATE INDEX idx_utterances_created ON utterances(created_at); Column Meaning and Source sessions.id UUID generated by the frontend with crypto.randomUUID() when session starts. sessions.target_language Plain language name chosen by user: "English", "Hindi", "Marathi". Appendix: Quick Reference — All Key Decisions Decision What Was Decided and Why Translation as core feature Live Language Translation promoted from stretch goal to the primary feature. It defines the product identity. Source language omitted always Whisper-1 auto-detects. No language parameter ever sent. This enables universal language support with zero UI friction. Two-model pipeline Whisper-1 for speech+detection, GPT-4o for translation. Neither model can do the full job alone. GPT-4o not Whisper translation Whisper translation endpoint only outputs English. GPT-4o supports 100+ target languages. 5-second chunk interval Balance between perceived latency (shorter feels more live) and accuracy (longer gives Whisper-1 more context). Memory storage for chunks Translation audio chunks are small (<2MB). Memory is faster and leaves no disk residue. End-of-session batch save Avoids 50+ Supabase calls during active session. Only completed sessions are persisted. Supabase service role key server-side Required for server-side writes. Bypasses RLS — MUST stay on backend only. Low temperature for GPT-4o (0.1) Translation must be accurate and consistent, not creative. Low temperature enforces this. Code-switching in GPT-4o prompt Rule 3 in the system prompt explicitly handles mixed-language speech. Critical for Indian language users. Vue.js frontend Reactive state management handles complex session lifecycle transitions cleanly. Whisper-1 primary model Chosen for SRT/VTT format support for the transcription feature. verbose_json used for translation (confidence data). Railway/Render deployment Auto-HTTPS, GitHub integration, free tier available, environment variables via dashboard. TRANSLATION_MODEL env var Allows switching between gpt-4o and gpt-4o-mini without code changes. Use mini during development.
